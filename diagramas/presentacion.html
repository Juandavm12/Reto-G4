Refactorización con Creator & SRP

Sistema de Biblioteca — Reto Técnico Grupo 4

Creator — GRASP SRP — SOLID

El Problema

La clase SistemaBiblioteca concentra 5+ responsabilidades y crea objetos que no le corresponden. Es una God Class. 

class SistemaBiblioteca:

    def registrar_usuario(self, nombre, email):
        usuario = Usuario(nombre, email)          # 1. Crea Usuario ✓
        tarjeta = Tarjeta(usuario.id)             # 2. Crea Tarjeta ✗ (debería ser Usuario)
        print(f"Guardando usuario y tarjeta en BD...")  # 3. Persistencia ✗
        print(f"Enviando email a {email}")              # 4. Notificación ✗
        return usuario

    def realizar_prestamo(self, usuario, libro):
        prestamo = Prestamo(usuario.id, libro.id) # 5. Crea Préstamo
        libro.disponible = False                   # 6. Modifica Libro directamente ✗
        print(f"Guardando préstamo en BD...")         # 7. Persistencia ✗
        print(f"Notificando a {usuario.nombre}")      # 8. Notificación ✗
        return prestamo

Violaciones de Creator (GRASP)

  • SistemaBiblioteca crea Tarjeta, pero no la contiene ni la usa directamente.
  • Según Creator, Usuario debería crear Tarjeta porque la contiene/agrega.

Violaciones de SRP (SOLID)

  • Persistencia en BD (guardar usuario, guardar préstamo).
  • Notificaciones (enviar email, notificar usuario).
  • Lógica de negocio (modificar estado de Libro).
  • Orquestación de flujos.
  • 4+ razones de cambio en una sola clase.

Los Principios

🔵 Creator — GRASP

"La clase B debería crear instancias de A si B contiene, agrega, registra, utiliza directamente o tiene los datos de inicialización de A."

En nuestro caso: Usuario contiene a Tarjeta, por lo tanto Usuario es el creador natural de Tarjeta.

🟣 SRP — SOLID

"Una clase debería tener una, y solo una, razón para cambiar."

SistemaBiblioteca cambiaba si cambiaba la BD, el email, la lógica de negocio o los modelos. Ahora cada responsabilidad vive en su propia clase.

Respuestas al Reto

1. ¿Cuántos objetos crea registrar_usuario()?
Crea 2 objetos: un Usuario y una Tarjeta. Ambos son instanciados directamente dentro del método.
2. ¿Quién DEBERÍA crear Tarjeta según Creator?
Usuario debería crear Tarjeta, ya que Usuario contiene/agrega a Tarjeta. Según el principio Creator de GRASP, la responsabilidad de creación recae en quien contiene, agrega, registra o usa directamente al objeto creado. En esta relación, la Tarjeta pertenece al Usuario.
3. ¿Cuántas responsabilidades tiene SistemaBiblioteca?
Tiene al menos 5 responsabilidades:
① Crear objetos del dominio (Usuario, Tarjeta, Prestamo)
② Persistencia en base de datos
③ Envío de notificaciones/emails
④ Modificar estado de otros objetos (Libro)
⑤ Orquestar flujos de negocio

Según SRP, debería tener una sola razón de cambio.
4. ¿Qué code smell ves aquí?
God Class / Blob — una clase que lo hace todo. También se aprecia Feature Envy al modificar directamente libro.disponible = False en vez de pedirle al propio Libro que gestione su estado. Hay acoplamiento fuerte y baja cohesión.

Cómo se Refactorizó

Paso 1 — Aplicar Creator: Usuario crea Tarjeta

✗ Antes
# En SistemaBiblioteca
usuario = Usuario(nombre, email)
tarjeta = Tarjeta(usuario.id)
✓ Después
# En Usuario.__init__
self.tarjeta = self._crear_tarjeta()

def _crear_tarjeta(self):
    return Tarjeta(self.id)

Paso 2 — Aplicar SRP: Extraer Repositorios

✗ Antes
# Todo dentro de SistemaBiblioteca
print(f"Guardando usuario y tarjeta en BD...")
print(f"Guardando préstamo en BD...")
✓ Después
# Clases separadas con responsabilidad única
class RepositorioUsuarios:
    def guardar(self, usuario): ...

class RepositorioPrestamos:
    def guardar(self, prestamo): ...

Paso 3 — Aplicar SRP: Extraer Notificador

✗ Antes
print(f"Enviando email a {email}")
print(f"Notificando a {usuario.nombre}")
✓ Después
class Notificador:
    def enviar_email(self, email): ...
    def notificar_usuario(self, usuario): ...

Paso 4 — Aplicar SRP: Libro gestiona su estado

✗ Antes
# Desde SistemaBiblioteca
libro.disponible = False
✓ Después
# Libro gestiona su propio estado
libro.marcar_no_disponible()

# Dentro de la clase Libro:
def marcar_no_disponible(self):
    self.disponible = False

Resultado Final

Antes

  • 1 clase con 5+ responsabilidades
  • SistemaBiblioteca crea Tarjeta (viola Creator)
  • Persistencia embebida en la lógica
  • Notificaciones embebidas en la lógica
  • Modifica objetos externos directamente
  • Alta acoplamiento, baja cohesión

Después

  • 8 clases, cada una con 1 responsabilidad
  • Usuario crea Tarjeta (Creator ✓)
  • Repositorios separados para persistencia
  • Notificador independiente
  • Libro gestiona su propio estado
  • Bajo acoplamiento, alta cohesión
